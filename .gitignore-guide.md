# .gitignore Configuration Guide

This document explains the .gitignore files configured for the Shipment Delay Analysis Application.

## Files Created

### 1. Root `.gitignore` (/.gitignore)
Covers the entire project including both client and server directories.

**Sections:**
- **Node Modules**: Excludes `node_modules/` and npm/yarn logs
- **Environment Variables**: Excludes all `.env` files (critical for security)
- **IDE & Editor**: Excludes VS Code, IntelliJ, and other editor files
- **Build & Dist**: Excludes build outputs and distribution files
- **Logs**: Excludes log files
- **OS Files**: Excludes OS-specific files (.DS_Store, Thumbs.db)
- **Testing**: Excludes coverage reports and test caches
- **Misc**: Excludes cache files and other temporary files

### 2. Client `.gitignore` (/client/.gitignore)
Specific to the React frontend application.

**Additional Sections:**
- **React Specific**: Environment files for different environments
- **Package Locks**: Excludes package-lock.json and yarn.lock
- **Vite/Turbopack**: Support for modern build tools if added later

### 3. Server `.gitignore` (/server/.gitignore)
Specific to the Node.js/Express backend application.

**Additional Sections:**
- **Server Specific**: Process files (.pid, .seed) and production env files

## What Gets Ignored

### 🔐 Security (Most Important)
```
.env                    # API keys, credentials
.env.local             # Local environment overrides
.env.*.local           # Environment-specific secrets
```

### 📦 Dependencies
```
node_modules/          # Installed packages
package-lock.json      # Dependency lock file
yarn.lock              # Yarn lock file
```

### 🏗️ Build Outputs
```
build/                 # React build output
dist/                  # Distribution files
.next/                 # Next.js output (if used)
```

### 🛠️ Development Tools
```
.vscode/               # VS Code settings
.idea/                 # IntelliJ settings
.DS_Store              # macOS files
Thumbs.db              # Windows files
```

### 📝 Logs & Caches
```
*.log                  # Log files
coverage/              # Test coverage reports
.eslintcache           # ESLint cache
.cache/                # Various caches
```

## What Should Be Committed

✅ **DO commit:**
- Source code (.js, .jsx, .css, .html)
- Configuration files (package.json, .babelrc, etc.)
- Documentation (README.md, guides)
- .gitignore itself
- Public assets (images, fonts)

❌ **DON'T commit:**
- .env files with secrets
- node_modules/
- Build outputs
- IDE configuration files
- OS-specific files
- Log files

## Environment Variables Setup

### For Development
Create `.env.local` in both client and server directories:

**server/.env.local:**
```
OPENAI_API_KEY=your_key_here
OPENAI_API_KEY_BACKUP=your_backup_key_here
OPENAI_API_BASE_URL=https://api.ai.kodekloud.com/v1
GEMINI_MODEL=google/gemini-3-flash-preview
PORT=5004
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_SECURE=false
EMAIL_USER=your_email@gmail.com
EMAIL_PASS=your_app_password
```

### For Production
Use environment variables provided by your hosting platform (Vercel, Heroku, etc.)

## Best Practices

1. **Never commit .env files** - Always use .gitignore
2. **Use .env.example** - Create a template file showing required variables
3. **Document secrets** - Keep a secure record of all API keys
4. **Rotate keys regularly** - Update API keys periodically
5. **Use different keys per environment** - Dev, staging, production keys should differ

## Verifying .gitignore Works

Check if files are properly ignored:
```bash
# List all files that would be committed
git status

# Check if a specific file is ignored
git check-ignore -v filename

# See what's in .gitignore
cat .gitignore
```

## Troubleshooting

### If .env is already committed:
```bash
# Remove from git (but keep locally)
git rm --cached .env

# Commit the removal
git commit -m "Remove .env from tracking"
```

### If node_modules is already committed:
```bash
# Remove from git
git rm -r --cached node_modules/

# Add to .gitignore
echo "node_modules/" >> .gitignore

# Commit
git commit -m "Remove node_modules from tracking"
```

## Summary

The .gitignore configuration ensures:
- ✅ Security: API keys and credentials are never exposed
- ✅ Cleanliness: Build artifacts and dependencies aren't tracked
- ✅ Efficiency: Repository size stays small
- ✅ Consistency: Team members have clean working directories
